# ERC-8004 Setup

BlockOps uses ERC-8004 to give agents an on-chain identity, public manifests, validation requests, and reputation updates. This turns each agent into a verifiable runtime unit rather than a UI-only record in a database.

## What We Implemented

- On-chain registration through the ERC-8004 Identity Registry.
- Reputation updates after verified execution.
- Validation requests linked to transaction results.
- Public agent manifests at `/agents/:id/manifest`.
- Structured execution logs exposed as `agent_log.json` data in runtime output.
- Registry discovery and trust-gated delegation scaffolding.

```bash
REPUTATION_REGISTRY_ADDRESS=0xa497e1BFe08109D60A8F91AdEc868ffdD1e0055c
VALIDATION_REGISTRY_ADDRESS=0xFab8731b8d1a978e78086179dC5494F0dbA1f6bE
SERVER_SIGNER_PRIVATE_KEY=your_backend_signer_private_key
ARBITRUM_SEPOLIA_RPC_URL=https://sepolia-rollup.arbitrum.io/rpc
```

Notes:

- `SERVER_SIGNER_PRIVATE_KEY` is used when an agent needs to be registered on-chain and the request does not include a user signer.
- The agent manifest endpoint can work without registration, but on-chain proof requires the registry addresses.

## Runtime Flow

The ERC-8004-aware runtime in `agentRuntime.js` follows this pattern:

1. Ensure the agent has an on-chain identity.
2. Execute the requested tool plan.
3. Verify transaction receipts when a tool returns a transaction hash.
4. Submit validation requests to the Validation Registry.
5. Update reputation through the Reputation Registry.
6. Export structured execution logs.

This data is surfaced back to the chat UI through `toolResults.runtime`.

## API Endpoints

Identity and manifest:

- `POST /agents/:id/register-on-chain`
- `GET /agents/:id/manifest`

Registry:

- `GET /agents/registry/discover`
- `GET /agents/:id/registry`
- `PUT /agents/:id/registry`

Audit and archived execution content:

- `GET /agents/:id/audit-logs`
- `GET /agents/:id/audit-logs/:logId/content`

## Multi-Agent Coordination

`agentCoordinator.js` provides the trust-gated handoff layer.

Current behavior:

- scan registered agents from the Identity Registry
- read `successRate` from the Reputation Registry
- pick the highest-scoring agent above the trust threshold
- return delegation metadata for the runtime

This gives the project a direct path toward multi-agent coordination without changing the core single-agent execution flow.

## How to Set It Up

1. Add the registry addresses and signer key to `backend/.env`.
2. Run the `on_chain_id` migration and the registry/audit migration in Supabase.
3. Start the backend.
4. Create an agent in the UI.
5. Register the agent on-chain from the UI or through `POST /agents/:id/register-on-chain`.
6. Open the manifest endpoint to confirm the agent is publicly describable.

## How to Verify It

1. Confirm the agent receives an `on_chain_id`.
2. Open `GET /agents/:id/manifest`.
3. Run a chat action that produces a tool result.
4. Confirm the chat UI shows ERC-8004 badges, verification data, or reputation data where applicable.
5. Inspect `toolResults.runtime.agent_log` and the registry or audit endpoints.

## Why ERC-8004 Matters Here

ERC-8004 gives BlockOps three important things:

- agents can be referenced outside the app
- execution history has a standard public shape
- trust and verification data can move with the agent identity
