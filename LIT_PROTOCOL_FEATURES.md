# BlockOps Hackathon Submission Guide

## Selected Track

This submission is for the **Lit Protocol V1 (Naga) SDK** track.

Our project demonstrates how autonomous AI agents can use **decentralized key management** instead of relying on user-exported private keys or always-on browser wallets. We use Lit Protocol to give users a **true seedless programmable wallet experience** while preserving compatibility with our earlier encrypted-key flow.

## One-Line Summary

**BlockOps** is an AI agent platform for blockchain operations where users can create agents, manage wallets, and execute onchain actions from the web app or Telegram. The flagship experience is a **Lit PKP wallet on Naga testnet** that is minted without ever exposing a seed phrase or raw private key.

## Problem We Solve

Most AI-powered crypto apps still have a serious wallet problem:

- users are asked to paste a private key into an app
- or they must keep a browser wallet open for every action
- or automation becomes custodial because the server stores signing credentials

That creates a bad user experience and weakens the trust model for autonomous agents.

For AI agents to become practical, they need wallets that are:

- non-custodial
- programmable
- usable across multiple interfaces
- safe for delegated signing and automation

## Why Lit Protocol

Lit Protocol is the core cryptographic layer that makes our approach possible.

We selected Lit because it gives us capabilities that traditional smart contracts and typical wallet integrations do not:

- **Distributed key generation**
  - PKPs are generated by Lit's decentralized node network, not by our frontend or backend
- **Programmable signing**
  - signing can be routed through Lit-backed identities instead of a locally stored EOA key
- **Programmable encryption**
  - our legacy path still protects imported EOAs with Lit Actions so raw private keys are never stored in plaintext
- **Cross-surface wallet identity**
  - the same Lit-backed wallet metadata can be reused in the web UI and in backend-driven Telegram automation

## What We Built With Lit Protocol

Our Lit integration has two layers:

### 1. Flagship PKP wallet flow

This is the main hackathon feature.

When a user opens the wallet setup modal, the recommended default tab is:

> **Create PKP Wallet**

That flow:

1. authenticates through our Lit server integration
2. mints a **Programmable Key Pair (PKP)** on **Lit Protocol Naga testnet**
3. stores only public wallet metadata in Supabase
4. uses that PKP later for transaction signing

What we store for PKP wallets:

- `wallet_type = 'pkp'`
- `wallet_address`
- `pkp_public_key`
- `pkp_token_id`

What we do **not** store:

- no PKP private key
- no seed phrase
- no raw key material generated by the app

### 2. Legacy encrypted-key compatibility flow

We also kept our existing wallet flow for compatibility with older users and demos.

That flow allows users to:

- import an existing EOA private key
- or generate a traditional private key inside the app

But instead of saving the raw secret, we use **custom Lit Actions** to:

- encrypt the key before storage
- store the encrypted payload as `lit:v1:...`
- decrypt only when the legacy signing path is explicitly used

This means the app now supports both:

- **PKP mode** for the true decentralized seedless experience
- **traditional mode** for migration-safe support of existing wallets

## Technical Architecture Overview

### High-level architecture

BlockOps has three main layers:

1. **Frontend**
   - Next.js application for user onboarding, agent creation, wallet setup, and chat-based blockchain operations

2. **Backend**
   - Node.js services for Telegram integration, transaction orchestration, AI workflow routing, and server-side PKP signing

3. **Persistence**
   - Supabase for user records, wallet metadata, and agent-linked state

### Wallet architecture

We now support two wallet types in the same `users` table:

- **PKP wallet**
  - `wallet_type = 'pkp'`
  - stores public PKP metadata only
  - signs with Lit SDK v8 on `naga-test`

- **Traditional wallet**
  - `wallet_type = 'traditional'`
  - stores a Lit-encrypted `lit:v1:` payload
  - decrypts through Lit Actions when needed

### Signing architecture

At runtime, every signing request branches on `wallet_type`:

- if `wallet_type = 'traditional'`
  - use the legacy Lit Action decrypt path
  - reconstruct the signer from the decrypted EOA key

- if `wallet_type = 'pkp'`
  - use the stored `pkp_public_key` and `pkp_token_id`
  - create a PKP-backed signer through the Lit SDK path
  - sign and broadcast without ever exposing PKP private key material

This hybrid routing lets us preserve backwards compatibility while making the PKP flow the main user experience.

### Network architecture

Our demo currently uses two networks for different responsibilities:

- **Lit Naga testnet**
  - PKP minting
  - Lit auth
  - Lit-backed signing infrastructure

- **Arbitrum Sepolia**
  - blockchain transaction execution in the current BlockOps app flows

So the Lit wallet layer and the execution chain are intentionally separated:

- Lit handles decentralized key management
- Arbitrum Sepolia is where the visible app transaction is broadcast today

## Exactly How We Use Lit Protocol

### PKP minting on Naga

We use Lit SDK packages in our server integration to mint PKPs on `naga-test`.

Core behavior:

- connect to Lit Naga nodes
- authenticate through a controller EOA
- mint a new PKP
- return `walletAddress`, `pkpPublicKey`, and `pkpTokenId`

This minting flow is implemented in:

- `frontend/lib/lit-server.ts`
- `frontend/app/api/lit/pkp/mint/route.ts`

### PKP-backed signing

After minting, we use the stored PKP metadata for signing.

We support PKP signing in both:

- the Next.js application
- the Node.js backend / Telegram flow

This is implemented in:

- `frontend/lib/lit-server.ts`
- `frontend/lib/lit-pkp.ts`
- `frontend/app/api/lit/pkp/sign/route.ts`
- `backend/services/litPkpService.js`

### Lit Actions for encryption and decryption

Our older wallet path uses custom Lit Actions:

- `ENCRYPT_ACTION_CODE`
- `DECRYPT_ACTION_CODE`

These actions are used to encrypt and decrypt a traditional EOA private key without saving it in plaintext in Supabase.

This logic is implemented in:

- `frontend/lib/lit-server.ts`
- `frontend/lib/lit-private-key.ts`
- `frontend/app/api/lit/private-key/encrypt/route.ts`
- `frontend/app/api/lit/private-key/decrypt/route.ts`

### Cross-platform wallet usage

A major part of the demo is that the same stored Lit-backed wallet can be used in two different product surfaces:

- **Web frontend**
  - wallet setup modal
  - contract interaction UI
  - agent chat transaction execution

- **Telegram/backend**
  - Telegram-linked user flow
  - backend transaction execution path
  - direct tool execution with PKP-aware routing

This proves the wallet is not just UI decoration. It is part of the application's actual autonomous execution model.

## Product Flow

### Main demo flow

1. User signs in to BlockOps
2. User opens the wallet setup modal
3. Default recommended tab is **Create PKP Wallet**
4. User clicks **Generate PKP**
5. App mints a PKP on Lit Naga testnet
6. App stores only public PKP metadata in Supabase
7. User triggers an agent or contract action
8. Signing routes through the PKP path
9. The same user can later use Telegram and the backend reuses the same PKP wallet metadata

### Fallback flow

If a user chooses **Import Existing Wallet**, we still support:

- importing a traditional private key
- Lit encryption before storage
- Lit decryption only during legacy signing

## Setup Instructions

### Prerequisites

- Node.js 20+
- Supabase project
- Lit Protocol configuration for Naga testnet
- one burner EOA to act as the Lit controller wallet
- Arbitrum Sepolia RPC access

### Environment variables

At minimum, configure these values in the frontend environment:

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY`
- `LIT_API_BASE_URL`
- `LIT_USAGE_API_KEY`
- `LIT_PKP_ID`
- `LIT_PKP_CONTROLLER_PRIVATE_KEY`
- `LIT_NAGA_HANDSHAKE_TIMEOUT_MS`
- `NEXT_PUBLIC_ARBITRUM_SEPOLIA_RPC_URL`
- `NEXT_PUBLIC_BLOCKCHAIN_BACKEND_URL`
- `NEXT_PUBLIC_AI_AGENT_BACKEND_URL`

Backend services also need the matching Supabase and Lit configuration if you want to test Telegram or server-side PKP signing.

### Database setup

Run the Supabase migration:

- `frontend/MIGRATION_FIX.sql`

This adds support for:

- `wallet_type`
- `pkp_public_key`
- `pkp_token_id`

### Install and run

Frontend:

```bash
cd frontend
npm install
npm run dev
```

Backend:

```bash
cd backend
npm install
npm run dev
```

AI workflow service:

- run the existing FastAPI workflow service used by the project
- the repository already contains `AI_workflow_backend/main.py` for this layer

## Testing Instructions

### Test the Lit PKP flow

1. Start the frontend and backend services
2. Sign in to the app
3. Open the wallet setup modal
4. Confirm the highlighted default option is **Create PKP Wallet**
5. Click **Generate PKP**
6. Confirm the success UI displays a PKP public key and token ID
7. Check Supabase and verify the row contains:
   - `wallet_type = 'pkp'`
   - `wallet_address`
   - `pkp_public_key`
   - `pkp_token_id`
8. Confirm there is no raw private key stored
9. Trigger a blockchain action from the web app
10. Verify the transaction uses the PKP signing path

### Test the legacy Lit encryption flow

1. Reopen the wallet modal
2. Switch to **Import Existing Wallet**
3. Import or generate a traditional EOA
4. Confirm the database stores a Lit-encrypted `lit:v1:` payload
5. Trigger a signing action
6. Verify the request uses the decrypt-and-sign path

### Test cross-platform PKP usage

1. Link the same user to the Telegram/backend flow
2. Trigger an action that requires signing
3. Verify the backend detects `wallet_type = 'pkp'`
4. Confirm the backend uses PKP signing instead of private key decryption
